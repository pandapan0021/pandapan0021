jinja2-模板对象

Template
(1) Tempalte类为核心模板对象。用来表示编译过的模板文件并用来渲染该模板文件。
(2) 通常模板对象是从Enviroment 类生成的，但Template 类提供了构造器来生成模板对象。
(3) 从Template类直接生成的模板对象包含一个指向临时环境的”environment”属性，所有由Template类直接生成的模板对象共享这个临时环境。
(4)__new__

args:

  1. source: 模板字符串。
  2. block_start_string(默认为 { % )
  3. block_end_string(默认为 % } )
  4. variable_start_string(默认为 { { )
  5. variable_end_string(默认为 } } )
  6. comment_start_string(默认为 { # )
  7. comment_end_string(默认为 # } )
  8. line_statement_prefix(None)
  9. line_comment_prefix(None)
  10. trim_blocks(False)
  11. lstrip_blocks(False)
  12. newline_sequence(默认为 \n )
  13. keep_trailing_newline(False)
  14. extensions
  15. optimized
  16. undefine
  17. finalize
  18. autoescape(此处默认为False，所以从字符串直接构造的模板对象不会自动转义，但是Environment生成的模板对象默认自动转义)
  19. enable_async(False)

函数体：
通过get_spontaneous_environment生成环境变量env，再通过env.from_string函数生成模板对象。

######(5)from_code
此函数从编译过的模板代码和globals参数创建模板对象，供loaders和environment调用。

######(6)from_module_dict
此函数通过模块构造模板对象，供module loadder调用

######(7)_from_namespace
由from_code与from_module_dict函数调用真正创建模板对象的函数

######(8)render/render_async
渲染函数，render_async需要支持await特性的python版本，目前还没有实现。

######(9)stream
同generate但是返回TemplateStream的实例

######(10)generate/generate_async
对于大规模的模板，此函数可以一条语句接着一条语句渲染并一段一段的生成。异步方法同样需要await支持(未实现)

从几个渲染函数看出，渲染的核心功能由Template.new_context函数与Template.root_render_func函数提供

######(11)new_context
为模板对象创建上下文环境，为Context类的实例。全局变量默认加入此上下文。

######(12)make_module/make_module_async
默认参数下和module方法一致。

######(13)module 调用 _get_default_module 返回self._module
由make_module 默认参数构造。

######(14)get_corresponding_lineno 返回行号

######(15)is_up_to_date 检测是否需要更新模板

######(16)debug_info 返回debug信息

sqlalchemy源码之Query

根据官方文档，Query类在平时应用中用户并不直接接触， 而是由session通过调用session.query()方法调用。
一、sqlalchemy.orm.session.py
在session的__init__方法中，可以看到参数query_cls的默认值为Query类。
接着在函数体中将session._query_cls赋值为query_cls,正常情况下都是使用默认值，及session._query_cls就是Query类。
session.query的定义为：

def query(self, *entities, **kwargs):
    return self._query_cls(entities, self, **kwargs)

可以看出session.query()的使命就是用他的参数得到一个Query的实例。

一般的用法是
session.query(MappedClass).filter……

可以看到传给实例化Query所用的参数就是用户定义的models。
那Query对这个MappedClass是怎么处理的呢

二、sqlalchemy.orm.query.py
Query.__init__中将self.session设置为初始化时的第二个参数就是之前已经实例化的session。之后处理传入的MappedClass参数。
对MappedClass参数的处理一直嵌套调用函数下面去掉函数调用整理成顺序处理

self._entities=[]
self._primary_entity = None
self.has_mapper_entities = False
for ent in util.to_list(entities):
    _QueryEntity(self, ent)

QueryEntity做了什么：
这是一个很神奇的类，会根据提供的初始化参数来改变自己。
通过\_new__方法使得自身在_MapperEntity，_Bundleentity, _ColumnEntity之间切换。通过源码可以看出正常提供MappedClass参数之后实例化的是_MapperEntity这个类。之后就是实例化_MapperEntity的过程：

def __init__(self, query, entity):
    if not query._primay_entity:
        query._primary_entity = self
    query._entities.append(self)
    query._has_mapper_entities = True
    self.entities = [entity]
    self.expr = entity
    ......

这时候_primary_entity还是None,所以设置_primary_entity为_MapperEntity的实例，接着在_entities列表中添加相同的_MapperEntity实例。可以看到当传入多个MappedClass时候，Query._primary_entity为传入的第一个Class实例化的_MapperEntity，Query._entities中存储的是所有的Class实例化的_MapperEntity。

设置完Query._primay_entity和Query._entities之后回到Query的实例化过程。

（Query类中）接着调用

self._set_entity_selectables(self._entities)

主要代码:

for ent in entities:

flask-redirect相关

在使用flask-login时，由login_required跳转到login_view完成登录之后可以由request.arg[‘next’]跳转回登录前的页面。
那不是由login_required产生的跳转怎么样主动重定向回之前的视图呢？
先看一下flask-login的做法：
源码flask_login/utils.py

def login_required(func):
    @wraps(func)
    def decorated_view(*args, **kwargs):
        if request.method in EXEMPT_METHODS:
 #  EXEMPT_METHODS = ['optionh']
            return func(*args, **kwargs)
        elif current_app.login_manager._login_disabled:
            return func(*args, **kwargs)
        elif not current_user.is_authenticated:
            return current_app.login_manager.unauthorized()
        return func(*args, **kwargs)
    return edcorated_view

忽略判断语句只看用户未登录时执行的动作，login_manager为LoginManager的实例。
源码flask_login/login_manager.py

request.url是的形如http://www.example.com/myapplication/π/page.html?x=y的当前完整路径。
可以看出flask_login做法就是把request.url做成next参数，查api可知request对于当前uri的属性一共有6个：
path
full_path
script_root
base_url
url

这些属性都是从werkzeug的BaseRequest类继承而来。
BaseRequest除了这几个uri相关的属性外 还有一个 CommonRequestDescriptorsMixin.referrer 表示发出请求的路径， 它们都是Request的父类。
所以要得到跳转前的rul可以使用flask-login的request.url也可使用request.referrer

flask源码

flask现在的源码太杂，先看简单的第一版源码。通过git checkout出。
只有一个文件(flask.py)，356行！！！！！！！

了解了wsgi基本原理之后，通过跟踪数据流的方式来看flask源码。
wsgi的application端需要提供一个可调用的对象，flask提供的可调用对象就是Flask类的实例。

先看flask程序的简单结构

from flask import Flask

app = Flask(__name__)

@app.route('/')
def index():
    pass

if __name__ == "__main__":
    app.run()

当构建Flask实例之前做了什么？
在导入模块是执行了

_request_ctx_stack = LocalStack()
current_app = LocalProxy(lambda: _request_ctx_stack.top.app)
request = LocalProxy(lambda: _request_ctx_stack.top.request)
session = LocalProxy(lambda: _request_ctx_stack.top.session)
g = LocalProxy(lambda: _request_ctx_stack.top.g)

具体细节先不管，可以简单理解为构造了一个本地栈，并生成本地上下文。
既然current_app, request, session, g都是在这时候生成的，为什么官方文档上说current_app,g为应用上下文，而request， session是请求上下文呢？

当我们通过Flask类产生实例对象时又做了什么？

self.view_functions = {}
self.error_handlers={}
self.request_init_funcs = []
self.reequest_shutdown_funcs = []

......设置static映射

简单来说就是就是产生一系列数据结构供以后使用。

route 函数干了什么

def route（self， rule， ××options）：
    def decorator(f):
        if 'endpoint' not in options:
            options['endpoint'] = f.__name__
        self.url_map.add(Rule(rule, **options))
        self.view_functions[options['endpoint']] = f
        return f
    return decorator

通过修饰器将修饰的函数注册到self.view_functions和self.url_map中。
endpoint 是一个key-argument，没见使用过，根据源码应该就是给修饰的函数取个别名，用于之后的映射。相当于django中url函数的第三个参数！！！

errorhandler, request_init(现在flask中为before_request), request_shutdown(现在为after_request)，提供相似的功能。

第一版flask并没有提供blueprint，但是通过上面的处理方法可以想象，
应该在register_blueprint的时候将blueprint的视图函数与规则注册从blueprint的结构中引入到app中。

准备就绪调用Flask的run()方法，通过werkzeug提供的run_simple启动一个简单的wsgi服务器来供调试时使用

程序完成之后wsgi怎么调用呢？
flask官方文档给出了部署方法以uwsgi为例

$ uwsgi -s /tmp/uwsgi.sock --manage-script-name --mount /yourapplication=myapp:app

最后/yourapplication=myapp:app就是告诉uwsgi，我的可调用对象是myapp模块中的名为app的flask实例

部署之后的数据流：

  1. wsgi服务器有request到来
  2. wsgi调用flask实例(app)

  3. 执行app.call()

    def __call__(self, environ, start_response):
    return self.wsgi_app(environ, start_response)

  4. call调用wsgi_app函数

    def wsgi_app(self, environ, start_response):
    _request_ctx_stack.push(_RequestContext(self, environ))
    try:
        rv = self.preprocess_request()
        if rv is None:
            rv = self.dispatch_request()
        response = self.make_response(rv)
        response = self.process_response(response)
        return response(environ, start_response)
    finally:
        _request_ctx_stack.pop()

下面看下wsgi_app的执行过程。

  1. 产生request上下文并且入栈

    _RequestContext根据environ设置request，又根据request设置session
    再重置g中的值，这就是为什么request和session是请求上下文的原因， 而g虽然每次请求都会重置但是它每次重置结果并不依赖于传入的environ,所以为应用上下文。

  2. 执行由request_init修饰器修饰的函数，
  3. 如果预处理函数产生输出怎么跳过剩下的处理过程直接构建response
  4. 如果预处理函数无输出怎么向下递交request给视图函数
  5. 构建response并返回
  6. request上下文出栈

再来看看response的构建过程。

def make_response(self, rv):
    if isinstance(rv, self.response_class):
        return rv
    if isinstance(rv, basestring):
        return self.response_class(rv)
    if isinstance(rv, tuple):
        return self.response_class(*rv)
    return self.response_class.force_type(rv, request.environ)

根据传入的不同结果选择行为， 最后都是返回一个response_class实例

flask源码--理解wsgi

PEP333-wsgi

PEP333

wsgi是什么？

定义了python web app 与 web servers 之间接口的协议

为自由而生（不受框架与服务器的限制而产生）

wsgi的两端

server or gateway 调用由application端提供的可调用的对象
application的行为就像一个函数， 当server端接到request则调用application并产生结果返回给server端。
下面为官方示例：

def simple_app(environ, start_response):
    """Simplest possible application boject"""
    status = '200 OK'
    response_header = [('Content-type', 'text/plain')]
    start_response(satus, response_headers)
    return ['Hello world!\n']

environ与start_response都在server调用时由server提供
server端示例：

request带来时处理过程：

       当request到来时，run_with_cgi开始运行
                 设置环境变量
                   判断协议
                调用application
               application端执行

             application返回data
               application结束
  server端把status，headers，data打印到标准输出

以上为官方示例的处理过程。

在server端与application之间还定义了一层中间件（middleware）组件。
中间件在整个系统中扮演着两种不同的角色。
对于server端它是表现的行为就像是一个application。
对于application端它的行为又像是一个server。
这意味着它同时实现了server端与application端的接口,并在同时遵循wsgi对于两端的限制。
中间件一般提供以下几个功能。

  • 根据url将request路由到不同的application。
  • 允许多个应用与框架在用一个过程中一起运行。
  • 通过在网络上传送request和response的方式加载均衡器或远程过程。
  • 执行内容后处理。

在存在中间件的系统中数据流向就像一个协议栈一样从server端一路向下直到到达app端然后再从app端一路向上回到server。

特别注意

  • environ和start_response只是调用application时使用的参数可以任意命名，没有特殊需求。
  • environ参数必须是python的内置dict类型
  • start_response必须包含两个位置参数和一个可选参数，并且必须被application以位置参数的形式调用
  • status是一个字符串r’\d{3}\s\w+’,response_headers是元组(header_key, header_value)的列表。
  • start_response只有在确认application调用成功之后才更新responese_headers
  • 。。。。。。
